Um para Um (1:1)
Cenário: Um Usuário tem um único Endereço.

Onde fica a FK? Na tabela de Endereços.

Lógica: Users.hasOne(Addresses) e Addresses.belongsTo(Users).
////////////////////////////////


Um para Muitos (1:N)
Cenário: Um Usuário pode cadastrar vários Jogos.

Onde fica a FK? Na tabela de Jogos (ela recebe o user_id).

Lógica: Users.hasMany(Games) e Games.belongsTo(Users).
////////////////////////////////


Muitos para Muitos (N:N)
Cenário: Muitos usuários favoritam muitos jogos.

Onde fica a FK? Em uma terceira tabela (tabela de junção), como a sua favorites.

Lógica: Users.belongsToMany(Games, { through: 'favorites' }).

Atendendo ao seu pedido, vamos recapitular as três propriedades principais do Sequelize (`attributes`, `where` e `include`).

Para ficar bem fácil de visualizar, imagine que a tabela de `Games` no seu banco de dados seja exatamente como esta planilha:

| id | name | console_id | city | user_id |
| --- | --- | --- | --- | --- |
| 1 | Call of Duty | 2 | São Paulo | 99 |
| 2 | God of War | 2 | Rio de Janeiro | 105 |
| 3 | FIFA 26 | 4 | São Paulo | 99 |

---

### 1. `attributes` (O que você quer Trazer)

Pense no `attributes` como o **seletor de colunas** da sua planilha. Por padrão, o Sequelize traz todas as colunas da tabela (`id`, `name`, `console_id`, `city`, `user_id`). Se você só precisa listar as cidades para o seu `<select>`, você não precisa gastar internet trazendo o resto.

```javascript
// Diz para o banco: "Olhe para a planilha e me devolva APENAS a coluna city"
attributes: ['city']

```

**O resultado seria apenas:** `['São Paulo', 'Rio de Janeiro', 'São Paulo']`.

---

### 2. `where` (O Filtro das Linhas)

Pense no `where` como o **filtro de linhas** da planilha (o famoso "funil"). Ele dita quais registros têm o direito de aparecer no resultado final baseado em condições.

```javascript
where: {
    city: 'São Paulo',           // 💡 Só traz as linhas onde a cidade for exatamente 'São Paulo'
    user_id: { [Op.ne]: 99 }     // 💡 Só traz as linhas onde o dono do jogo NÃO SEJA o usuário 99
}

```

---

### 3. `include` (O Relacionamento / Juntar Planilhas)

Imagine que você tem uma segunda planilha chamada `Consoles` que guarda os nomes reais dos videogames (`1 = Nintendo Switch`, `2 = PlayStation 4`). Na planilha de `Games`, você só salvou o número `console_id`.

O `include` serve para ir até essa segunda planilha, buscar o nome bonito do console e colar o dado junto com o jogo correspondente. É o equivalente ao `VLOOKUP` / `PROCV` do Excel ou ao `JOIN` do SQL.

```javascript
include: [{
    model: Consoles,             // 💡 Vai até a tabela de Consoles
    attributes: ['name_console'] // 💡 Pega apenas o nome do videogame lá dentro
}]

```

---

### 🚀 Juntando tudo na sua nova rota de Cidades

Como você quer uma rota independente que traga as cidades sem repetição para o seu `useState`, veja como as três propriedades trabalham juntas com o `group` para fazer exatamente o que você quer:

```javascript
// No seu CitiesController.js
export async function getAvailableCities(req, res) {
    try {
        const cities = await Games.findAll({
            // 1. attributes: só queremos o texto da cidade
            attributes: ['city'], 
            
            // 2. where: filtramos para não trazer cidades de jogos do próprio usuário logado
            where: {
                user_id: { [Op.ne]: ActiveUser } 
            },
            
            // 3. group: remove as duplicadas! Se houver 50 "São Paulo", o grupo junta em uma só
            group: ['Games.city'], 
            
            // (Não precisamos de 'include' aqui, pois a cidade já está gravada direto no jogo)
        });

        // Retorna a lista limpa: [{ city: "São Paulo" }, { city: "Rio de Janeiro" }]
        return res.json(cities); 
        
    } catch (error) {
        return res.status(500).json({ error: "Erro ao buscar cidades" });
    }
}

```

Guarde essa colinha! Com o `attributes` você escolhe as colunas, com o `where` você filtra as linhas, e com o `include` você traz dados de outras tabelas relacionadas.
